# Skill: Writing `.clang-uml` configuration files

This document teaches an LLM (or a human) how to author `.clang-uml`
configuration files for `clang-uml` — a tool that generates UML class,
sequence, package and include diagrams directly from C++, C and Objective-C
source code, using the Clang compiler frontend.

Diagrams can be emitted in PlantUML (`.puml`), MermaidJS (`.mmd`), GraphML
and JSON formats.

## Prerequisites

1. The project must have a `compile_commands.json` compilation database
   (e.g. generated by CMake with `-DCMAKE_EXPORT_COMPILE_COMMANDS=ON`,
   or by Bear, Meson, etc.).
2. The configuration file is YAML and by default is named `.clang-uml`,
   placed in the project's top-level directory.
3. Run `clang-uml` in that directory to generate diagrams, then render
   e.g. with `plantuml -tsvg diagrams/*.puml` or
   `mmdc -i diagram.mmd -o diagram.svg`.

Useful commands:

```bash
clang-uml --init                      # create initial .clang-uml
clang-uml -l                          # list diagrams in config
clang-uml -n <name>                   # generate a single diagram
clang-uml -g mermaid                  # select generator
                                      # (plantuml|mermaid|json|graphml)
clang-uml --progress                  # show progress bars
clang-uml --dump-config               # print effective config
clang-uml --print-from -n <name>      # list valid 'from' locations
clang-uml --print-to -n <name>        # list valid 'to' locations
```

## Overall file structure

```yaml
# Top-level (global) options — inherited by all diagrams
compilation_database_dir: _build
output_directory: diagrams
diagrams:
  diagram_name_1:
    type: class            # class | sequence | package | include
    # ... diagram options
  diagram_name_2:
    type: sequence
    # ... diagram options
```

Rules:

* Each key under `diagrams` is the diagram name and also the output file
  basename.
* Common options set at the top level are inherited by every diagram
  unless the diagram overrides them.
* All relative paths (`glob`, `output_directory`,
  `compilation_database_dir`, `paths` filter) are resolved relative to
  the config file's directory, unless `relative_to` is set.

## Top-level options

| Option                     | Meaning                                          |
|----------------------------|--------------------------------------------------|
| `compilation_database_dir` | Directory containing `compile_commands.json`     |
| `output_directory`         | Where diagram files are written                  |
| `diagrams`                 | Map of diagram definitions                       |
| `add_compile_flags`        | Flags appended to every compile command          |
| `remove_compile_flags`     | Flags removed (string or regex `r:`)             |
| `query_driver`             | Compiler to query for system include paths       |
| `allow_empty_diagrams`     | Do not fail on empty diagrams                    |
| `debug_mode`               | Add source location comments to output           |
| `user_data`                | Arbitrary data usable in Jinja templates         |
| `diagram_templates`        | Reusable Jinja diagram templates                 |

Example for fixing missing system headers (common on macOS or
cross-compilation):

```yaml
query_driver: arm-none-eabi-g++
add_compile_flags:
  - -I/opt/my_toolchain/include
remove_compile_flags:
  - r: "-m.*"
```

## Options common to all diagram types

| Option              | Meaning                                              |
|---------------------|------------------------------------------------------|
| `type`              | `class`, `sequence`, `package` or `include`          |
| `glob`              | Translation units to parse (see below)               |
| `include`/`exclude` | Diagram content filters (see below)                  |
| `using_namespace`   | Render names relative to this namespace              |
| `title`             | Diagram title                                        |
| `comment_parser`    | `plain` (default) or `clang`                         |
| `filter_mode`       | `basic` (default) or `advanced` (for `anyof`/`allof`)|
| `relative_to`       | Root for relative paths                              |
| `generate_links`    | Hyperlinks per element (Jinja patterns)              |
| `plantuml`/`mermaid`| `before:`/`after:` lists of verbatim directives      |
| `graphml`           | `notes:` map of element name -> list of notes (not available for sequence diagrams) |
| `generate_metadata` | Emit clang-uml metadata in output                    |
| `debug_mode`        | Emit source-location comments                        |

### glob — selecting translation units

`glob` selects which translation units (from `compile_commands.json`) are
parsed. Files matching glob but absent from the compilation database are
ignored. Omitting `glob` parses the whole database (slow on large
projects — always constrain it when possible).

```yaml
# Simple form — all patterns are inclusive; regex via `r:` key
glob:
  - src/dir1/*.cc
  - src/**/*.cpp
  - r: ".*test.*cpp$"

# Explicit form with include/exclude
glob:
  include:
    - src/**/*.cc
  exclude:
    - src/main.cc
```

### include / exclude filters

Both `include` and `exclude` accept the same filter structure. An element
is kept if it matches the `include` filters and does not match `exclude`.
Values are exact qualified names or regexes written as `r: 'pattern'`.

| Filter            | Applies to           | Values                                              |
|-------------------|----------------------|-----------------------------------------------------|
| `namespaces`      | all                  | `ns1::ns2` or `r: '.*detail.*'`                     |
| `modules`         | all                  | C++20 module names, e.g. `mod1.mod2`                |
| `elements`        | all                  | Qualified names, regexes, or typed entries          |
| `element_types`   | class                | `class`, `enum`, `concept`, `method`, `function`, `function_template`, `lambda` |
| `paths`           | all                  | Files/dirs where elements are declared              |
| `context`         | class                | Elements related to a given class (see below)       |
| `relationships`   | class                | `inheritance`, `composition`, `aggregation`, `ownership`, `association`, `instantiation`, `friendship`, `dependency` |
| `subclasses`      | class                | Subclasses of given class (incl. itself)            |
| `parents`         | class                | Base classes of given class                         |
| `specializations` | class                | Template specializations/instantiations             |
| `dependants`      | class                | Classes depending on given class                    |
| `dependencies`    | class                | Classes the given class depends on                  |
| `access`          | class                | `public`, `protected`, `private`                    |
| `module_access`   | all                  | `public`, `private`                                 |
| `method_types`    | class                | `constructor`, `destructor`, `assignment`, `operator`, `defaulted`, `deleted`, `static` |
| `callee_types`    | sequence             | `constructor`, `assignment`, `operator`, `defaulted`, `static`, `method`, `function`, `function_template`, `lambda`, `cuda_kernel`, `cuda_device` |
| `anyof` / `allof` | all (advanced mode)  | Logical OR / AND over nested filters                |

Typical filter usage:

```yaml
include:
  namespaces:
    - myproject
exclude:
  namespaces:
    - myproject::detail
  elements:
    - r: '.*test.*'
    - type: method            # typed element filter
      name:
        r: '.*::(get|set).*'  # exclude all getters/setters
  access:
    - private
```

Typed `elements` filter — the `type` key can be: `any`, `function`,
`function_template`, `class`, `enum`, `method`, `member`, `concept`,
`package`, `objc_method`, `objc_member`, `objc_protocol`,
`objc_category`, `objc_interface`.

Context filter (class diagrams) — include only elements related to a
class, with optional radius, direction and relationship kinds:

```yaml
include:
  context:
    - ns1::MyClass            # simple: radius 1, any direction
    - match:
        radius: 3
        pattern: ns1::OtherClass
        direction: inward     # inward | outward | any
        relationships:
          - inheritance
          - aggregation
```

Logical operators (require `filter_mode: advanced` in the diagram):

```yaml
filter_mode: advanced
include:
  anyof:
    subclasses:
      - ns1::nsA::A1
    namespaces:
      - ns2::nsB
```

## Class diagrams (`type: class`)

Minimal example:

```yaml
diagrams:
  my_classes:
    type: class
    glob:
      - src/*.cc
    using_namespace: myproject
    include:
      namespaces:
        - myproject
```

Class-diagram-specific options:

| Option                                    | Meaning                                            |
|-------------------------------------------|----------------------------------------------------|
| `generate_method_arguments`               | `full` (default), `abbreviated`, `none`            |
| `generate_packages`                       | Group elements into UML packages                   |
| `package_type`                            | `namespace` (default), `directory`, `module`       |
| `include_relations_also_as_members`       | Render relationship members inside class box (default `true`) |
| `generate_template_argument_dependencies` | Dependencies from template args (set `false` to declutter) |
| `skip_redundant_dependencies`             | Drop dependencies implied by other relations (default `true`) |
| `generate_concept_requirements`           | Render C++20 concept requirements (default `true`) |
| `member_order`                            | `lexical` or `as_is`                               |
| `group_methods`                           | Group methods by category                          |
| `using_module`                            | Render module names relative to this module        |
| `type_aliases`                            | Map of type name substitutions                     |
| `relationship_hints`                      | Override relationship kind for template args       |
| `layout`                                  | Layout hints per element                           |

Relationship hints and layout hints:

```yaml
relationship_hints:
  std::tuple: aggregation      # all args of std::tuple -> aggregation
  ns1::some_template:
    default: association
    2: composition             # arg at index 2 -> composition
layout:
  ClassA:
    - up: ClassB
    - left: ClassC
  ClassD:
    - together: [ClassE, ClassF]
  ClassX:
    - row: [ClassY1, ClassZ1]
    - column: [ClassY2, ClassZ2]
```

Common class diagram recipes:

```yaml
# Inheritance hierarchy of a base class
include:
  subclasses:
    - myproject::BaseClass
  relationships:
    - inheritance

# Context diagram of a single class (for its docs page)
include:
  context:
    - match:
        radius: 2
        pattern: myproject::MyClass

# Public API only
include:
  access:
    - public

# Declutter: no dependencies, no template arg dependencies
generate_template_argument_dependencies: false
exclude:
  relationships:
    - dependency
```

## Sequence diagrams (`type: sequence`)

Sequence diagrams REQUIRE at least one location constraint (`from`,
`to`, or `from_to`) specifying where message chains start and/or end.
Values are full function/method signatures or regexes; methods need
fully qualified names.

```yaml
diagrams:
  main_sequence:
    type: sequence
    glob:
      - src/main.cc
    include:
      paths:
        - src
    exclude:
      namespaces:
        - std
    from:
      - function: "main(int,const char**)"
    # or regex:
    # from:
    #   - function:
    #       r: "main.*"
    # or end-point / range constraints:
    # to:
    #   - function: "myproject::Engine::run()"
    # from_to:
    #   - [ function: "myproject::A::start()",
    #       function: "myproject::B::finish()" ]
```

To discover exact signatures, run:
`clang-uml --print-from -n <diagram>` or `--print-to`.

Sequence-diagram-specific options:

| Option                                          | Meaning                                        |
|-------------------------------------------------|------------------------------------------------|
| `from` / `to` / `from_to`                       | Location constraints (`start_from` is a deprecated alias of `from`) |
| `combine_free_functions_into_file_participants` | Group free functions per source file           |
| `inline_lambda_messages`                        | Hide lambda participants, inline their calls   |
| `generate_return_types`                         | Render return types on return messages         |
| `generate_return_values`                        | Render return expressions on return messages   |
| `generate_condition_statements`                 | Render `if`/`while` conditions in alt/loop     |
| `generate_message_comments`                     | Render code comments as notes (needs `-fparse-all-comments`) |
| `generate_method_arguments`                     | `full`, `abbreviated`, `none`                  |
| `generate_method_argument_names`                | Also render argument names                     |
| `fold_repeated_activities`                      | Render repeated activities only once           |
| `message_comment_width` / `message_name_width`  | Truncation widths                              |
| `participants_order`                            | Explicit left-to-right participant order       |

To include code comments or manually injected calls
(`// \uml{call ns::B::bbb()}`), add at the top level:

```yaml
add_compile_flags:
  - -fparse-all-comments
```

## Package diagrams (`type: package`)

Render namespaces, directories or C++20 modules as packages with their
dependencies:

```yaml
diagrams:
  architecture:
    type: package
    glob:
      - src/**/*.cc
    package_type: namespace   # namespace | directory | module
    include:
      namespaces:
        - myproject
```

Use `package_type: directory` for C projects or directory-structured
codebases; `using_module`/`package_type: module` for C++20 modules.

## Include diagrams (`type: include`)

Visualize `#include` dependencies between project files:

```yaml
diagrams:
  includes:
    type: include
    glob:
      - src/*.cc
    relative_to: .              # render paths relative to this dir
    generate_system_headers: false  # true adds directly-included
                                    # system headers (can be huge)
    generate_packages: true     # false = flat nodes with full paths
    include:
      paths:
        - src
        - include
```

## Hyperlinks (`generate_links`)

Make SVG diagram elements clickable, e.g. linking to GitHub:

```yaml
generate_links:
  link: "https://github.com/user/repo/blob/{{ git.commit }}/\
{{ element.source.path }}#L{{ element.source.line }}"
  tooltip: "{{ element.name }}"
```

Available Jinja context includes `element.*` (name, source.path,
source.line, ...) and `git.*` (branch, revision, commit, toplevel).

## Diagram templates

Reusable Jinja templates for generating many similar diagram configs:

```yaml
diagram_templates:
  parents_hierarchy_tmpl:
    description: Generate class parents inheritance diagram
    type: class
    template: |
      {{ diagram_name }}:
        type: class
        include:
          parents: [{{ class_name }}]
          namespaces: [{{ namespace_name }}]
          relationships:
            - inheritance
```

Built-in templates: `parents_hierarchy_tmpl`, `subclass_hierarchy_tmpl`,
`class_context_tmpl`. Use them via:

```bash
clang-uml --add-diagram-from-template class_context_tmpl \
  --template-var diagram_name=myclass_context \
  --template-var class_name=myproject::MyClass \
  --template-var namespace_name=myproject
```

External YAML fragments (e.g. diagram definitions or the
`diagram_templates` node) can be included via
`include!: path/to/file.yml`.

## Complete annotated example

```yaml
compilation_database_dir: _build
output_directory: docs/diagrams
add_compile_flags:
  - -fparse-all-comments
generate_method_arguments: none      # inherited by all diagrams
generate_links:
  link: "https://github.com/user/repo/blob/{{ git.commit }}/\
{{ element.source.path }}#L{{ element.source.line }}"
  tooltip: "{{ element.name }}"
diagrams:
  core_classes:
    type: class
    title: Core model classes
    glob:
      - src/model/*.cc
    using_namespace: myproject
    generate_packages: true
    package_type: namespace
    include:
      namespaces:
        - myproject::model
    exclude:
      access:
        - private
      elements:
        - r: '.*test.*'
    plantuml:
      before:
        - left to right direction
  main_flow:
    type: sequence
    glob:
      - src/main.cc
    include:
      paths:
        - src
    exclude:
      namespaces:
        - std
    from:
      - function: "main(int,const char**)"
  packages_overview:
    type: package
    glob:
      - src/**/*.cc
    include:
      namespaces:
        - myproject
  include_graph:
    type: include
    glob:
      - src/**/*.cc
    include:
      paths:
        - src
        - include
```

## Checklist and common pitfalls

When generating a `.clang-uml` file for a project:

1. Locate `compile_commands.json` and set `compilation_database_dir`
   accordingly (often `_build`, `build` or `debug`).
2. Identify the project's top-level namespace(s) — use them in
   `using_namespace` and `include.namespaces`. For C projects (no
   namespaces), use `paths` filters and `package_type: directory`.
3. Constrain `glob` to the translation units needed for each diagram —
   parsing everything is slow on large projects. Remember `glob` matches
   only files present in the compilation database.
4. Regular expressions in filters must be written as `r: 'pattern'`
   maps, not bare strings, and match against fully qualified names.
5. Sequence diagrams need a `from`, `to` or `from_to` constraint;
   function signatures must match exactly (use `--print-from` /
   `--print-to` to discover them) or use regexes.
6. Prefer several small, focused diagrams (context, inheritance,
   package overview) over one huge diagram.
7. `anyof`/`allof` filters require `filter_mode: advanced`.
8. If Clang reports missing headers (e.g. `'stddef.h' file not found`),
   add `query_driver`, or `add_compile_flags`/`remove_compile_flags`.
9. To exclude standard library noise, add `exclude.namespaces: [std]`.
10. Test the config with `clang-uml -n <diagram_name> --progress` and
    inspect warnings about empty diagrams or unmatched glob patterns.
